# Quick Task 260707-36d: Fix admin.js renderAdmin default team names Summary

Fixed a first-render bug where `renderAdmin()` clobbered `buildRegistrationBlock()`'s prefilled default team names with an empty string, and updated the default names so all 5 pass server-side length validation.

## What Changed

**Task 1 — Gate textarea restore (commit 9e6536e):**
In `renderAdmin()`, the CR-02 restore block (`newInput.value = preservedValue`) ran unconditionally. On the very first render `existingInput` is `null`, so `preservedValue` was `''`, overwriting the freshly prefilled defaults. Changed the guard from `if (newInput)` to `if (existingInput && newInput)` so the restore only runs when there was a prior textarea to preserve from. First render keeps the defaults; re-renders still preserve in-progress typing/focus/selection (CR-02 intact).

**Task 2 — Update defaultTeamNames (commit 04ae87b):**
In `buildRegistrationBlock()`, replaced the 4-entry array (2 of which exceeded the server's 40-char `MAX_TEAM_NAME_LENGTH`) with 5 entries, all ≤40 chars:
1. "Els primers no sempre seran els ultims" (38 chars — shortened from 49)
2. "Segundas filas siempre fueron buenas" (unchanged)
3. "Tercera fila NO té perque ser negatiu" (37 chars — shortened from 52)
4. "Los ultimos de las filas" (unchanged)
5. "Conectamos con Evovisión" (new, owner-requested)

All 5 ≤ MAX_TEAMS (6), so the batch passes `isValidTeamNamesPayload` when registered as-is.

## Verification

- Task 1: `admin.js` contains `if (existingInput && newInput)` — confirmed via node.
- Task 2: node script confirms exactly 5 names, all ≤40 chars — `OK: 5 names, all <=40 chars`.

None — plan executed as written. Note: the plan's `<objective>` prose refers to "4 default team names" in places, but the `must_haves.truths` and Task 2 action explicitly specify **5** names. Implemented 5 per the authoritative task spec.
