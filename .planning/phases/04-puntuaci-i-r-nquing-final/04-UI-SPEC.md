# Phase 4 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

**Scope of this contract:** the NEW UI surfaces introduced by scoring/ranking, the shared award-ceremony sequence that precedes the ranking reveal, plus three changes to the existing per-phase "Finalitzar" affordance:

1. **Team final results screen** — a new full-screen state (`state.phase === 'results'` / a terminal `finished` flag) shown to **all** teams simultaneously when the Admin fires ADMIN-07. Shows the **global ranking + global percentage of every team**, but the **sub-check detail is visible only for the current team's own work** — never another team's (D-10, D-11).
2. **Admin partial mini-ranking** — an **Admin-only** compact ranking rendered in the Admin panel each time a game phase closes (HTML→CSS, CSS→JS). Teams never see it (D-12, D-13). Uses the same scoring pipeline with not-yet-played phases counting as 0.
3. **Award ceremony sequence (D-14)** — a **shared full-screen animated overlay** that plays **simultaneously on every screen (all teams AND the Admin)** the instant ADMIN-07 fires, *before* any ranking is shown: an animated 5→0 countdown, a 3-second hold on 0, then a reverse-order ("and the winner is…") reveal of the ranking, ending in full-screen confetti. It **precedes** and then transitions into the existing surfaces above (each screen's own ranking) — it does not alter them.
4. **"Finalitzar" affordance changes** (GAME-08 retrofit): HTML button gains a **100%-correctness gate** (D-07), the CSS button is **removed** (D-08), JS has **no button** (D-09). The Admin CTA at the last phase becomes **"Finalitzar i Mostrar Resultats"** (ADMIN-07) — and this is now the trigger that launches the award ceremony on all screens (D-14).

**Inheritance:** this contract EXTENDS the Phase 1–3 UI-SPECs. All existing tokens in `src/client/shared/tokens.css`, the LIGHT theme, the 60/30/10 surface split, phase color-coding (`--phase-html/css/js`), the typography roles, the spacing scale, the monospace code-label pattern, the motion tokens, the frozen overlay, the Admin `.team-grid`/`.team-card`, the `.btn`/`.btn-accent` system, and the `.confirm-dialog` pattern are reused verbatim. **This phase adds no new design tokens** — every new surface maps onto existing `var(--*)` tokens, with a **single explicitly-scoped exception**: the celebratory "chillón" countdown colors of the award ceremony (D-14), which are deliberately *outside* the token system, live only inside the ephemeral `.ceremony-overlay`, and are unmounted before any ranking screen renders (see Color §Award ceremony palette exception). Executor MUST reference existing tokens for every non-ceremony surface, never hard-coded literals (Phase 1–3 token strategy).

**Audience driver (rectora):** unchanged — students with zero prior experience, in a 15–20 min live classroom dynamic. The results screen is a **pedagogical debrief** (SCORE-05), not a leaderboard shaming ritual: a team's own concrete misses stay private to its own screen (D-10), the ranking shows only names + global % publicly (D-11), and a missed sub-check is rendered as **neutral/incomplete, never as a red error** (project no-frustration ethos, consistent with Phase 2/3 "state, never error color").

### Award ceremony palette exception (D-14) — one-time, sanctioned break of 60/30/10

The countdown numbers (5, 4, 3, 2, 1, 0) each render in a **different vivid/"chillón" color**, deliberately outside the inherited LIGHT palette. This is the **only** place in the entire product where a saturated multi-color palette is permitted.

**Why the exception is justified here specifically:**
- **Non-informational / purely celebratory.** These colors carry **no state, no meaning, no data**. Unlike every other surface (where color encodes phase identity, pass/miss, or provisional-ness), the countdown color is pure spectacle — a "party" beat marking the transition from work to celebration. The 60/30/10 discipline exists to keep *informational* UI calm and scannable; a 5-second confetti-style flourish is a different rhetorical register and the restraint rule simply does not apply to it.
- **Ephemeral and full-bleed.** Each color exists for ~1 second, on a full-screen overlay, and the whole sequence is gone in well under 10 seconds. It never coexists with informational UI (the ranking is not yet drawn while the numbers play).
- **Pedagogically on-purpose.** For zero-experience students in a 15–20 min live dynamic, the loud countdown is the emotional payoff that makes the debrief land — consistent with "celebrar el moment" (D-14 discretion).

**Chillón countdown colors** — literal, ephemeral sequence values (NOT design tokens, NOT reused, NOT added to `tokens.css`; they are ceremony data, like scores/percentages):

| Number | Color | Note |
|--------|-------|------|
| 5 | `#FF1E56` | vivid pink-red |
| 4 | `#FF9F1C` | vivid orange |
| 3 | `#FFD60A` | vivid yellow |
| 2 | `#2EC4B6` | vivid teal |
| 1 | `#3A86FF` | vivid blue |
| 0 | `#C77DFF` | vivid purple |

Rendered at Display scale (see Typography), centered, one at a time, on a neutral ceremony backdrop (`--color-bg`, or an optional dimmed scrim — implementer's choice). Confetti particles at the end MAY reuse this same six-color set.

**Containment invariant (must hold):** these colors appear **only** inside `.ceremony-overlay`, **only** during the D-14 sequence. They **MUST NOT bleed** into the ranking screen, the rank rows, the percentage bars, the sub-check icons, the Admin mini-ranking, or any other state. The overlay is removed from the DOM when the reveal completes; the ranking screen it transitions into keeps the fully neutral inherited palette (own-row emphasis stays neutral, misses stay `--color-muted`, passes stay `--status-connected`). No countdown color persists anywhere after the ceremony ends.

## Motion Contract

Reuses the inherited motion tokens — `--motion-snap: 150ms`, `--motion-ease: cubic-bezier(0.2,0,0.2,1)`. **No new motion tokens.** All motion is decorative; every surface is fully usable with motion disabled.

| Interaction | Read | Spec |
|-------------|------|------|
| **Results screen appears** | "Els resultats han arribat" | The ranking column fades/rises in over `--motion-snap` `--motion-ease` (mirrors the `.js-rule` enter). Rows may stagger subtly; not required |
| **Percentage bar fill** | Score magnitude | Each `.rank-bar` fill animates its width from 0 → % over `--motion-snap` on first paint |
| **Own global % (Display)** | The team's headline number | Static by default. An optional count-up is allowed but must land on the exact value; skip under reduced motion |
| **Admin mini-rank on phase close** | "Here's where teams stand now" | Appears instantly on the phase-close broadcast; no entrance animation required (Admin ops surface, clarity over flourish) |
| **Award ceremony (D-14)** | "The results are about to be announced — celebrate" | Full-screen overlay sequence; exact timing below. This is the one deliberately-heavy, celebratory animation in the product |

### Award ceremony timing (D-14) — exact values for the planner

The ceremony is a single scripted timeline. These are **literal sequence durations in ms** (ceremony data, not motion tokens — do not add to `tokens.css`). The whole thing plays identically on every team screen and the Admin screen, driven by one ADMIN-07 broadcast so they stay in lockstep.

| # | Step | Duration (ms) | Behavior |
|---|------|--------------|----------|
| 1 | Countdown numbers **5, 4, 3, 2, 1** | **1000 each → 5000 total** | Each number appears centered at Display scale in its chillón color, then over its 1000ms window does the **zoom-in + fade-out** "coming at you and past the viewer" motion (`scale` grows past 1 while `opacity` → 0). One number per second (1s cadence) — the animation fills the full second; the next number begins as the previous exits. |
| 2 | **0** static hold | **3000** | `0` appears centered in its color and holds **static — no zoom** for 3s (the dramatic pause). |
| 3 | **0** exit | **800** | `0` performs the same zoom-in + fade-out exit. |
| 4 | Ranking reveal (reverse order) | **~3600** (see pacing) | On `0`'s exit the ranking reveals **last place first → winner last**. |
| 5 | Confetti | **~4000** overlap/tail | Full-screen particle burst fires as the #1 row lands, then fades. |

**Reverse-reveal pacing (design discretion, D-14 → kept deliberately simple, low implementation cost):**
- Rows reveal **bottom rank first** (last place), one at a time, at a **fixed 600ms stagger**.
- A single **1200ms dramatic pause** is inserted **before the #1 (winner) row** — the only bespoke beat; everything else is a uniform interval. This gives the "…i el guanyador és…" swell without per-row hand-tuning.
- Each row uses the **existing enter treatment** (fade/rise over `--motion-snap` `--motion-ease`, same as `.rank-row` first paint) — no new easing.
- Worked example (6 teams): 6th @0ms, 5th @600, 4th @1200, 3rd @1800, 2nd @2400, **[+1200 pause]**, 1st @3600ms. For 4 teams it is proportionally shorter. The winner row landing is the confetti trigger.
- **Total ceremony ≈ 5000 + 3000 + 800 + ~3600 ≈ 12.4s**, with the confetti tail finishing by ~16–17s. Comfortably inside the celebratory budget for a 15–20 min session — do NOT lengthen further.

**Confetti implementation note (technical hint, not a visual spec):** implement as a lightweight **dependency-free** effect — either a short CSS-keyframe particle shower (a few dozen absolutely-positioned colored divs animating `translateY`/`rotate`) or a tiny one-off `<canvas>` requestAnimationFrame loop; do **not** pull in a confetti library (no heavy dependency for a ~4s one-shot).

**Reduced motion:** wrap all of the standard animations above in `@media (prefers-reduced-motion: no-preference)` (same pattern as `.timer-display` critical pulse and the Phase-2/3 animations). With reduced motion, the ranking, bars, and percentage render at their final values **instantly** — no count-up, no fill animation. **For the award ceremony specifically:** under `prefers-reduced-motion: reduce`, skip the zoom/fade and the confetti motion entirely — the sequence collapses to a calm degrade (either show each countdown number as a static 1s color frame with no scaling, or skip straight to the final ranking) and the ranking renders at its final state with no stagger. The ceremony is purely celebratory, so dropping it costs no information. Functionality and legibility fully preserved.

## Layout Contract

### Award ceremony overlay (all screens, D-14)

Full viewport, **rendered above everything** (both the team app and the Admin panel), on ADMIN-07. Single centered stage: one countdown number at a time, geometrically centered on both axes; the reveal ranking uses the same centered capped column as the ranking it precedes. The overlay owns the whole screen while it plays (no split, no controls, no interaction) and is torn down on completion, handing each screen to its already-rendered post-finalize view (team → results screen; Admin → final ranking). It plays identically and simultaneously on every screen because it is driven by one ADMIN-07 broadcast, not per-client timers.

### Team results screen

Full viewport, LIGHT theme, single centered capped column (`max-width: 640px`), `--space-3xl` vertical page padding — no split, no preview iframe (the game is over).

Top → bottom:
1. **Results heading** (Heading) — "Classificació".
2. **Ranking list** — all 4–6 `.rank-row`s, descending by global %, winner (#1) marked with the `Trophy` icon, the viewing team's row neutrally emphasized. Each row shows name + `.rank-bar` + global %.
3. **`--space-2xl` separation.**
4. **Own outcome** — the viewing team's own global % in Display ("El teu resultat"), then the **own sub-check detail** grouped by phase (HTML / CSS / JS) with phase-badge headers and pass/miss item lists. **Global % only, no per-phase % shown to the team** (D-11).

The screen is **read-only**: no buttons, no next action on the team side — the Admin owns the session lifecycle.
