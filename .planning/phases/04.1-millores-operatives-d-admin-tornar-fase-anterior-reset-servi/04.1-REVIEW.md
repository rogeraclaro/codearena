# Phase 04.1: Code Review Report

**Reviewed:** 2026-07-06T00:00:00Z
**Depth:** standard
**Files Reviewed:** 10
**Status:** issues_found

## Summary

Reviewed the "Fase anterior" / server-reset admin tooling plus the PM2 process-supervisor
infra added in this phase, and the incidental CSS live-overlay + default-team-names diffs
that landed in the same file set. The server-side half of the new work (`gameState.previousPhase`,
the `ADMIN_PREV_PHASE`/`ADMIN_RESET_SERVER` handlers, `test/prevPhase.test.js`) is solid: admin-room
re-validation, `durationMs` validation, and `doneAt`/timer semantics all match the documented
decisions and are exercised by real socket-level tests.

The two **critical** findings both sit in `src/client/admin.js`'s team-registration flow (the
"pre-fill default team names" feature added in this phase): the pre-existing input-preservation
logic silently wipes the new defaults on first paint, and — independently — two of the four
hardcoded default names exceed the server's 40-character name limit, so even after fixing the
first bug, clicking "Registrar" against the unedited defaults would silently fail with zero
admin-facing feedback. Given the pre-filled defaults exist specifically to save a few seconds of
typing during a live 15–20 minute classroom session, both defects defeat the feature's entire
purpose and should block ship.

Several warnings cover a dead broadcast event, a missing `finished` guard on the new
previous-phase mutation path, stale admin UI state after going back a phase, and PM2 config
robustness gaps relevant to the "reset is always available" emergency-tool framing.

## Critical Issues

### CR-01: Default team-names pre-fill is wiped immediately by the existing input-preservation logic

**File:** `src/client/admin.js:707-756`
**Issue:**
`buildRegistrationBlock()` (lines 505-515) creates the `#team-names-input` textarea and sets
`textarea.value = defaultTeamNames.join('\n')`. But `renderAdmin()`, which calls
`buildRegistrationBlock()`, snapshots whatever was in the *previous* textarea (`preservedValue`,
captured at line 708 **before** the new textarea exists) and then unconditionally overwrites the
brand-new textarea with that snapshot after mounting it:

```js
// admin.js:707-711 — captured BEFORE the new (defaulted) textarea exists
const existingInput = document.getElementById('team-names-input');
const preservedValue = existingInput ? existingInput.value : '';
...
// admin.js:745-756 — unconditionally restores the OLD snapshot onto the NEW textarea
const newInput = document.getElementById('team-names-input');
if (newInput) {
  newInput.value = preservedValue;   // '' on first render — wipes the just-set defaults
  ...
}
```

On the very first `session:full-state` render there is no pre-existing textarea, so
`preservedValue` is `''`. Two lines later, the freshly-created textarea (which the registration
block just filled with the four default team names) gets its `.value` forced back to `''` — all
within the same synchronous call, so the admin never sees the defaults at all. The feature this
phase added is functionally dead on arrival; the panel always renders with an empty textarea.

This is the same logic path that (correctly) preserves in-progress typing across unrelated
`session:full-state` re-renders (CR-02 in the original comment numbering) — it just wasn't
updated to account for the new default-value case.

**Fix:** Only restore the preserved value when a previous textarea actually existed (i.e. skip
the restore on the very first mount, so the freshly-set defaults survive):

```js
const newInput = document.getElementById('team-names-input');
if (newInput && existingInput) {
  newInput.value = preservedValue;
  if (hadFocus) {
    newInput.focus();
    if (selStart !== null && selEnd !== null) {
      newInput.setSelectionRange(selStart, selEnd);
    }
  }
}
```

### CR-02: Two of the four hardcoded default team names exceed the server's name-length limit

**File:** `src/client/admin.js:509-515` (defaults) and `src/server/socketHandlers.js:11,27-36` (validation)
**Issue:**
`socketHandlers.js` enforces `MAX_TEAM_NAME_LENGTH = 40` and rejects the **entire** batch if any
single name is too long:

```js
// socketHandlers.js:27-36
function isValidTeamNamesPayload(names) {
  return (
    Array.isArray(names) &&
    names.length > 0 &&
    names.length <= MAX_TEAMS &&
    names.every(
      (name) => typeof name === 'string' && name.trim().length > 0 && name.length <= MAX_TEAM_NAME_LENGTH,
    )
  );
}
```

The new defaults in `admin.js` are:

| Name | Length |
|---|---|
| `Els primers no sempre tenen perquè ser els últims` | **49** |
| `Segundas filas siempre fueron buenas` | 36 |
| `Que siguem de tercera fila NO té perquè ser negatiu!` | **52** |
| `Los ultimos de las filas` | 24 |

Two of the four exceed 40 characters. If CR-01 above were fixed and the teacher clicked
"Registrar" against the unedited, pre-filled defaults (the exact scenario the feature exists to
support), `isValidTeamNamesPayload` would reject the whole payload — `registerSocketHandlers`
returns silently with no error emitted back to the client. Meanwhile the client-side handler
already cleared the textarea and assumes success:

```js
// admin.js:521-529
registerBtn.addEventListener('click', () => {
  const names = textarea.value.split('\n').map((name) => name.trim()).filter((name) => name.length > 0);
  if (names.length === 0) return;
  socket.emit('admin:register-teams', { names });
  textarea.value = '';   // cleared unconditionally, before any server ack
});
```

Net effect: teacher clicks "Registrar" on the defaults, the textarea empties, no teams appear in
the grid, and there is no error message anywhere — a confusing dead end in the middle of a live,
time-boxed class with no obvious recovery path other than guessing what went wrong.

**Fix:** Shorten all four defaults to ≤40 chars (or raise `MAX_TEAM_NAME_LENGTH` deliberately), and
stop optimistically clearing the textarea / assume success without server confirmation, e.g.:

```js
const defaultTeamNames = [
  'Els primers no sempre són els últims',
  'Segundas filas siempre fueron buenas',
  'Tercera fila NO és sinònim de negatiu!',
  'Los ultimos de las filas',
];
```

and/or add a client-side length check with a visible inline error before emitting, so an
oversized name never gets silently swallowed.

## Warnings

### WR-01: `SERVER_RESTARTING` is broadcast but has no listener anywhere in the client

**File:** `src/server/socketHandlers.js:227-237`, `src/server/events.js:42`
**Issue:** The reset handler's own comment describes `SERVER_RESTARTING` as "avís amable abans
que el procés surti" (a friendly notice before the process exits):

```js
// socketHandlers.js:231
io.to('session').emit(EVENTS.SERVER_RESTARTING);
setTimeout(() => { io.close(); process.exit(0); }, 150);
```

Neither `src/client/admin.js` nor `src/client/client.js` (nor any other file in the repo) has a
`socket.on(EVENTS.SERVER_RESTARTING, ...)` / `socket.on('server:restarting', ...)` listener. The
event is emitted into the void — teams and the admin get no visual feedback at all when Reset is
pressed; screens just disconnect and reconnect via Socket.io's own auto-reconnect. Either the
client-side "friendly notice" was never implemented, or this broadcast is dead code.
**Fix:** Add a listener on both `admin.js` and `client.js` that shows a brief non-blocking
"Reiniciant el servidor…" banner/overlay when `SERVER_RESTARTING` arrives, or remove the
broadcast/comment if no UI is intended.

### WR-02: `previousPhase` has no `finished` guard, unlike the admin UI that gates the button

**File:** `src/server/gameState.js:321-326`, `src/server/socketHandlers.js:207-217`
**Issue:** `admin.js` only renders the "Fase anterior" button when `!finished` (line 433:
`if (state.phase && state.phase !== 'html' && !finished)`), and the code comments claim the
invariant "mai un cop mostrats els resultats" (never once results are shown). But neither
`gameState.previousPhase()` nor the `ADMIN_PREV_PHASE` socket handler re-validates
`state.finished` server-side — every other admin mutation in this file explicitly re-validates
room membership server-side per the "V4: never trust the client" convention used throughout, but
this particular invariant (no going back after finalize) is enforced only by hiding a button.
A stray duplicate emit, a race between two admin tabs, or a modified client could call
`admin:prev-phase` after `ADMIN_FINALIZE_GAME`, silently moving `state.phase` backward and
re-enabling `setCssValue`/`setJsRules` mutation for any team that hadn't already pressed
"Finalitzar" for that phase — even though the game is nominally over and a final ranking is
already frozen and displayed.
**Fix:** Add `if (state.finished) return false;` at the top of `previousPhase()` (mirrors the
idempotency guard already used in `finalizeGame()`), matching the invariant the admin UI already
assumes.

### WR-03: Admin's partial-ranking display goes stale after "Fase anterior"

**File:** `src/client/admin.js:23-24, 719-728, 787-791`
**Issue:** `partialRanking`/`partialClosedPhase` are module-level state set only by the
`ADMIN_PARTIAL_RANKING` event, which `ADMIN_NEXT_PHASE`'s handler emits when a phase is closed
(by design, `ADMIN_PREV_PHASE` deliberately does not emit it — see the comment in
`socketHandlers.js:200-206`). The render logic keeps showing the last partial ranking "fins que
arriba el rànquing final" (comment at admin.js:651-654), with no path to invalidate it. After the
admin uses the new "Fase anterior" button to reopen, say, the CSS phase, the panel keeps
displaying "Rànquing parcial — fase CSS tancada" even though CSS is once again the *active*,
editable phase — misleading the teacher into thinking that phase's scoring is final when it is
not.
**Fix:** Clear (or visually distinguish/expire) `partialRanking`/`partialClosedPhase` when
`ADMIN_PREV_PHASE` succeeds — e.g. have the server also emit a "ranking now stale" signal on
`previousPhase`, or have the admin client clear its local `partialRanking` whenever the received
`session:full-state.phase` no longer matches `partialClosedPhase`'s "next" phase.

### WR-04: PM2 `max_restarts: 30` conflicts with the "always-available emergency tool" framing of Reset

**File:** `ecosystem.config.cjs:13-25`
**Issue:** The Reset button is documented (`admin.js:452-457`) and implemented
(`socketHandlers.js:219-237`) as "Sempre disponible ... eina d'emergència" with no built-in
throttling. But `ecosystem.config.cjs` caps PM2 at `max_restarts: 30`; once that ceiling is
crossed (e.g. a genuinely rough class session where the teacher resets repeatedly to recover from
different problems), PM2 stops reviving the process and it stays down — silently, with no
signal surfaced anywhere in this codebase (no admin UI message, no server log consumed by
anything user-facing) that the "always available" emergency tool has stopped being available.
**Fix:** Either raise/remove `max_restarts` for this specific "operator-triggered restart" use
case, or explicitly document the 30-reset ceiling for operators, since the two currently
contradict each other.

### WR-05: `ecosystem.config.cjs` has no explicit `cwd`, so `pm2 start`/boot-time resurrection is directory-dependent

**File:** `ecosystem.config.cjs:13-25`
**Issue:** `script: 'server.cjs'` is a relative path with no accompanying `cwd`. PM2 resolves
relative scripts against the directory the `pm2 start` command was invoked from (or, for
`pm2 startup`/`pm2 resurrect` on boot, whatever working directory the PM2 daemon itself was
started/dumped from) — not necessarily the project root. Given this project explicitly calls out
VPS/Nginx/PM2 production robustness as a hard constraint (per CLAUDE.md), an implicit-cwd config
is a foot-gun for exactly the boot-time/"pm2 resurrect after VPS reboot" scenario PM2 is being
used for.
**Fix:** Add an explicit `cwd: __dirname` (available since this is a `.cjs` file) to the app
config.

**File:** `src/client/admin.js:30`, `src/client/client.js:38`
**Issue:** `const PHASE_LABELS = { html: 'HTML', css: 'CSS', js: 'JS' };` is defined identically
in both files. The rest of the codebase is explicit about avoiding this pattern (`EVENTS` in
`events.js`, the vocabulary tables in `shared/robotTemplate.js`) specifically so there's a single
source of truth to edit.
**Fix:** Move `PHASE_LABELS` into a shared module (e.g. `src/shared/robotTemplate.js` or a new
small `src/shared/phases.js`) and import it from both `admin.js` and `client.js`.
