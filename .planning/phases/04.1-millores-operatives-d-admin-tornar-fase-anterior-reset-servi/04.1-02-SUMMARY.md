## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug / Rule 3 - Blocking] PM2 fork mode silently broke the ESM auto-start guard — not the anticipated `ERR_REQUIRE_ESM`**
- **Found during:** Task 3 (boot checkpoint verification)
- **Issue:** `RESEARCH.md` Pitfall 1 anticipated that PM2 might fail to load the ESM entrypoint with an explicit `ERR_REQUIRE_ESM` error. Instead, `pm2 start ecosystem.config.cjs` pointed directly at `src/server/index.js` succeeded silently: `pm2 status` showed the process `online` with 0 restarts, but `curl http://localhost:3000/` returned connection-refused and `pm2 logs` showed **no output at all** — not even the startup `console.log`. Root cause, confirmed via a temporary debug line: PM2 fork mode executes the script through its own internal wrapper (`.../pm2/lib/ProcessContainerFork.js`), which becomes `process.argv[1]`. This silently makes `index.js`'s `isMainModule` guard (`process.argv[1] === fileURLToPath(import.meta.url)`) always `false` under PM2, so `startServer()` is never invoked — no crash, no error, no log, just a process that looks healthy but never listens.
- **Fix:** Created `server.cjs` at the repo root — a one-file CJS launcher that dynamically imports `src/server/index.js` and calls the exported `startServer()` directly, bypassing the `isMainModule` guard entirely (this file never checks `argv[1]`, so it is unaffected by PM2's wrapper rewrite). Updated `ecosystem.config.cjs`'s `script` field from `'src/server/index.js'` to `'server.cjs'`. Verified no test file depends on `index.js`'s implicit auto-start (`grep` confirmed every `test/*.test.js` imports `startServer` and calls it explicitly with `startServer(0)`), so this change is isolated to the PM2 boot path with zero impact on the test suite or `npm run dev`/`npm run server`/`npm start` (which still work as before, since `node src/server/index.js` sets `argv[1]` correctly and the guard still fires as intended in that path).
- **Files modified:** `ecosystem.config.cjs`, `server.cjs` (new)
- **Verification:** After the fix, `pm2 start ecosystem.config.cjs` → `pm2 logs` showed the clean line `CodeArena server listening on http://localhost:3000 (via PM2/server.cjs)` with no errors; `curl http://localhost:3000/` returned `404` (Express responding, not connection-refused); `kill -9` on the PM2-managed pid was followed by an automatic revive within ~2s (restart counter incremented, port responded again) confirming the `autorestart` mechanism Plan 03's Reset feature depends on.
- **Committed in:** `5308fac`
