# Phase 04.1 Plan 03: Reset servidor (Admin) Summary

**Admin-only ADMIN_RESET_SERVER socket event + handler that broadcasts a friendly SERVER_RESTARTING notice then kills the Node process via `io.close()`+`process.exit(0)` for PM2 to revive clean, wired to an always-visible destructive «Reset» button in the Admin panel.**

## Accomplishments
- `EVENTS.ADMIN_RESET_SERVER` (`'admin:reset-server'`, client→server, admin-only) and `EVENTS.SERVER_RESTARTING` (`'server:restarting'`, server→client) added to `events.js` as the single source of truth for the reset protocol.
- Admin-only handler in `socketHandlers.js` mirrors `ADMIN_FINALIZE_GAME`'s shape: `socket.rooms.has('admin')` is the first check (fail-closed, T-04.1-03 — a non-admin client cannot kill the shared process), then broadcasts `SERVER_RESTARTING` to `'session'`, then after a 150ms `setTimeout` calls `io.close()` (clean disconnect so clients auto-reconnect) and `process.exit(0)` (PM2 revive = full in-memory state wipe, D-04). No `SESSION_FULL_STATE` is emitted — this is a process-lifecycle event, not a game-state change, per the RESEARCH.md anti-pattern.
- `index.js` verified (not edited): `process.exit(0)` terminates the whole process, so there is no surviving interval to leak regardless of whether `httpServer`'s `close` event fires first.
- Admin panel gets a «Reset» button in `buildControlBar`, always visible (D-05 — an emergency tool, never gated by `state.phase` or `finished`), destructive-styled (`btn btn-destructive`). Clicking it awaits `showConfirm('Segur que vols reiniciar el servidor?', 'Sí, reinicia', true)` and, on confirmation, emits `EVENTS.ADMIN_RESET_SERVER` with no payload.

## Decisions Made
- Followed the plan's exact handler shape (mirror of `ADMIN_FINALIZE_GAME`) without adding a second confirmation layer or a manual reconnection handler on the client — `socket.io-client`'s default `reconnection: true` already covers post-restart reconnection (verified in RESEARCH.md Assumption A5).
- Did not modify `index.js`: the plan explicitly instructed "VERIFICA, NO editis a menys que calgui" and `process.exit(0)` makes any interval-leak concern moot since the entire process (and its event loop) disappears — there's nothing left to run a leaked `setInterval` callback.
- Reset button always renders (no `if` guard), unlike the "Fase anterior" button from Plan 01 which is conditionally shown — this matches D-05's explicit requirement that Reset be an emergency tool available at any point, not tied to game phase.
