# Pitfalls Research

**Domain:** Real-time gamified classroom webapp (Socket.io game, drag & drop editor, live single-classroom usage, auto-scoring)
**Researched:** 2026-07-01
**Confidence:** MEDIUM (web search cross-checked against official Socket.io docs and MDN; no curated/Context7 docs available for this ecosystem mix)

## Critical Pitfalls

### Pitfall 1: Team identity tied to `socket.id` instead of an application-level session token

**What goes wrong:**
A team refreshes their browser (F5) or their WebSocket drops for a moment — mid-game — and the server treats them as a brand-new, unregistered client. They lose their team name, their phase, their progress, and possibly get dropped into the lobby while the rest of the class has moved on.

**Why it happens:**
`socket.id` is regenerated by Socket.io on every reconnection, including a full page refresh. Developers who naively key their server-side game state map (`teams[socket.id] = {...}`) find that the key vanishes the instant the underlying transport reconnects, even though it's "the same browser tab reconnecting." Socket.IO v4.1+ ships a Connection State Recovery feature, but it only survives *brief* disconnects (default ~2 minutes) and does not survive a server restart or a long gap — not a substitute for real identity.

**How to avoid:**
Issue each team a durable session token (random string) on registration, store it in `localStorage` (survives refresh) and send it in the Socket.io `auth` payload on every connect/reconnect. Key all server-side game state by this token, not by `socket.id`. On `connection`, look up existing state by token and re-sync the client (current phase, remaining time, their current board state) rather than starting fresh.

**Warning signs:**
- Server code uses `socket.id` as a map key for anything that must survive more than one request.
- No `localStorage`/cookie write happens at team registration time.
- Manual test: register a team, refresh the tab — if the team has to re-register, this pitfall is present.

**Phase to address:**
Foundational "real-time core / state sync" phase — must be solved before phase mechanics are built on top of it, since every phase's client assumes identity persistence.

### Pitfall 2: Client-side countdown instead of server-broadcast absolute end time

**What goes wrong:**
Each team's timer visibly drifts from the others (and from the projected class screen). Teams see different remaining times, "Time's up!" fires at different moments per browser, and the admin's "+1 minute" or "Pause/Resume" controls desync client displays from the server's actual phase-ending logic — a team might keep working after the server has already auto-advanced the phase, or a slow tab shows extra time it doesn't actually have.

**Why it happens:**
The natural-seeming implementation is `setInterval(() => secondsLeft--, 1000)` running independently on each client. Each browser tab's timer drifts based on its own clock, tab throttling (background tabs slow `setInterval`), and network latency in receiving the initial "start" event. Elapsed-time-based sync (server sends "X seconds have passed") compounds this drift over a 15-20 minute session.

**How to avoid:**
Server is the single source of truth: it computes and broadcasts an absolute `phaseEndsAt` (epoch ms), not a countdown. Every client computes `remaining = phaseEndsAt - Date.now()` on each tick — this self-corrects regardless of when the message arrived or how much drift/jitter occurred. "+1 minute" simply adds 60000ms to `phaseEndsAt` server-side and rebroadcasts. "Pause" stores `remainingAtPause` and unsets the deadline; "Resume" recomputes a fresh `phaseEndsAt = now + remainingAtPause`. The server (not any client) is what actually enforces the phase transition when the deadline passes — client display is purely cosmetic.

**Warning signs:**
- Search codebase for `setInterval` combined with `secondsLeft -=` or similar decrementing logic on the client.
- Any client-side code that independently decides "time's up" and calls a phase-transition function without confirming with the server.
- Admin's "Pausar/Reprendre" button only pauses the client's local `setInterval` rather than telling the server.

**Phase to address:**
"Nucli temps real" / Admin panel timer phase — this is core plumbing shared by every subsequent phase (HTML/CSS/JS), so it must be right before phases are built.

### Pitfall 3: Duplicate sockets / zombie connections after reconnect or repeated refresh

**What goes wrong:**
A student who nervously hits F5 twice, or whose laptop briefly loses Wi-Fi and Socket.io auto-reconnects, ends up with two (or more) live sockets registered to the same team. The team then receives duplicate events, duplicate score submissions, or the admin's team-count/progress view shows phantom extra teams.

**Why it happens:**
Socket.io's default reconnection logic creates a new connection attempt before the old one has necessarily been cleaned up server-side; the old socket isn't always torn down synchronously. Combined with Pitfall 1 (no stable team-token join logic), a server that does `teams[token] = socket` on every `connection` event without explicitly disconnecting any *previous* socket for that same token will end up broadcasting to two channels for one team.

**How to avoid:**
On the server's `connection` handler, look up any existing socket already registered for that team token; if found, forcibly `.disconnect(true)` the old socket (or move it out of any rooms) before registering the new one. Treat "join" as a replace operation, never an append. Use Socket.io rooms keyed by team-token (not by raw socket.id) so `io.to(teamToken).emit(...)` naturally targets exactly one live connection.

**Warning signs:**
- Admin's team-status view occasionally shows more connections than the 4-6 registered teams.
- Team receives an emitted event twice on the client console.
- No explicit `oldSocket.disconnect()` call anywhere near the `connection`/`join` handler.

**Phase to address:**
Same "Nucli temps real" phase as Pitfall 1 — identity and connection lifecycle must be designed together.

### Pitfall 4: WebSocket upgrade silently falls back to (or gets stuck on) HTTP long-polling behind Nginx

**What goes wrong:**
In local development everything is instant. After deploying behind the VPS's Nginx reverse proxy, the game feels laggy, admin actions to teams take a visible beat to arrive, or the connection drops every ~60 seconds and silently reconnects (causing visible stutter or state hiccups) — but nothing "errors" outright, so it's hard to diagnose in a live classroom.

**Why it happens:**
Nginx does not proxy WebSocket upgrade headers by default. Socket.io transparently falls back to HTTP long-polling when the upgrade fails, which *works* but has materially higher latency and periodic reconnect churn — exactly the kind of subtle degradation that's easy to miss in a quick manual test but painful during a live, time-pressured 15-minute session. Additionally, if `proxy_read_timeout`/`proxy_send_timeout` are shorter than Socket.io's `pingInterval + pingTimeout`, Nginx kills the connection out from under Socket.io, forcing needless reconnects.

**How to avoid:**
Nginx location block for the Socket.io path must set `proxy_http_version 1.1;`, `proxy_set_header Upgrade $http_upgrade;`, `proxy_set_header Connection "upgrade";`, and set `proxy_read_timeout`/`proxy_send_timeout` comfortably above Socket.io's default ping settings (default pingInterval 25s + pingTimeout 20s = 45s; set proxy timeouts to 60s+). Confirm in production (not just localhost) that the browser dev tools Network tab shows a `101 Switching Protocols` / `websocket` transport, not stuck on `polling`. Because this project's deployment target is exactly "VPS + Nginx," this is not a hypothetical concern — verify it before the first live classroom run, not on presentation day.

**Warning signs:**
- Browser dev tools Network tab shows repeated polling requests (`transport=polling`) instead of a single upgraded `transport=websocket` connection once deployed.
- Latency noticeably higher in production than local dev with no other explanation.
- Occasional full reconnect/blip roughly every 20-60 seconds under sustained use.

**Phase to address:**
Deployment/production-readiness phase — but the Nginx config should be drafted and smoke-tested early (even against a staging subdomain) rather than discovered the day of the live class, since there is zero margin to debug this live in front of students.

### Pitfall 5: Auto-scoring compares style/DOM values by naive string equality instead of normalized values

**What goes wrong:**
A team correctly sets `background-color: red` but the Plana Model was authored as `#ff0000`, or a team writes `rgb(255, 0, 0)` with different spacing than the reference — the scoring engine marks it wrong despite being visually and functionally identical. This is a *fairness* bug: students did the right thing and got penalized by a scoring implementation detail, which is worse than a bug that's merely visually broken, because it's invisible and erodes trust in the exercise.

**Why it happens:**
`getComputedStyle` does normalize declared colors to `rgb()`/`rgba()` format regardless of source syntax, which actually helps — but pitfalls remain: `transparent` may be returned as the literal string `"transparent"` rather than `rgba(0,0,0,0)` in some contexts; whitespace/format differences (`rgb(0,0,0)` vs `rgb(0, 0, 0)`) can break naive string comparison; shorthand vs. longhand properties (e.g., `border` vs. `border-color`/`border-width`/`border-style`) resolve to different computed values; default/inherited values (an unset property inheriting from a parent) may not match what a "correct" explicit declaration produces even when visually identical.

**How to avoid:**
Never compare computed style strings directly. Parse every color to numeric RGBA components (regardless of source format) before comparing, with an explicit tolerance where relevant (e.g., anti-aliasing artifacts are not a concern here since these are declared styles, but rounding on some browser color-manipulation functions can occur). For layout/spacing properties compare numeric pixel values, not raw strings. Decide up front, explicitly, which properties count toward scoring (e.g., background-color, color, font-size) and write a small, unit-tested `compareComputedValue(prop, expected, actual)` function per property type rather than generic string equality — this is exactly the kind of "small function, well tested" surface area worth investing test time in, since it directly determines whether students perceive the game as fair.

**Warning signs:**
- Scoring code does `expected === actual` on any style-derived string.
- No normalization/parsing step exists before comparison.
- No unit tests for the scoring comparator using equivalent-but-differently-formatted inputs (hex vs rgb vs named color for the same color).

**Phase to address:**
Auto-scoring phase (HTML/CSS/JS comparison against Plana Model) — should include this as an explicit acceptance criterion: "scoring treats `red`, `#ff0000`, and `rgb(255,0,0)` as equivalent."

### Pitfall 6: HTML5 native Drag and Drop API chosen for the block editor, causing inconsistent/frustrating snap behavior

**What goes wrong:**
The Fase HTML drag & drop feels sluggish or inconsistent: drop targets don't highlight reliably while dragging, the "snap to nearest valid slot" behavior the project explicitly requires (to avoid frustration for non-technical students) is hard to implement cleanly, and behavior varies between browsers/input devices used in the classroom (mouse vs. trackpad vs. any touch-capable classroom laptops).

**Why it happens:**
The native HTML5 Drag and Drop API is built entirely on mouse events, not Pointer Events — it does not fire at all from touch input on Chrome/Firefox for Android and has known cross-browser inconsistencies in `DataTransfer`/`setDragImage` handling even on desktop. It also does not support custom "closest valid slot" snapping out of the box — you're fighting the browser's own drag ghost-image and auto-scroll behavior rather than controlling the interaction. Since this project's whole pedagogical premise depends on drag & drop feeling immediate and forgiving ("bloc mal deixat torna al calaix o encaixa a l'espai vàlid més proper"), a shaky implementation directly undermines the core value proposition.

**How to avoid:**
Use a Pointer Events-based drag library (e.g., `interact.js`, `dnd-kit`, or `SortableJS`) rather than the native `draggable` attribute + `dragstart`/`dragover`/`drop` events. These libraries use `setPointerCapture` and `elementsFromPoint` to give full programmatic control over drop-target detection, snapping, and visual feedback (including a "snap back to tray" animation), and they behave consistently across all pointer input types the classroom might use. This also future-proofs against any touch-screen classroom hardware without a rewrite.

**Warning signs:**
- Code uses `draggable="true"` plus `ondragstart`/`ondrop` native attributes.
- No consistent "snap to nearest slot" visual feedback during drag, only on drop.
- Any manual browser testing reveals the drag "ghost" image looks different across Chrome/Firefox/Edge.

**Phase to address:**
Fase HTML (drag & drop block editor) phase — decide the library before writing block-editor UI code; retrofitting later is a rewrite, not a patch.

### Pitfall 7: Student-generated content preview iframe is not actually sandboxed

**What goes wrong:**
Even though there's "no free code writing" per the project's design (blocks, sliders, dropdowns — not a raw code editor), the live preview pane still renders team-constructed HTML/CSS/JS. If that preview iframe isn't properly isolated, a bug in the block-to-HTML generation (or an unexpected input combination) could let generated content read/manipulate the parent page's DOM, cookies, or the Socket.io connection object itself — turning a pedagogical tool into an accidental security hole, and in the worst case letting one team's preview interfere with the admin page if it's rendered in the same context.

**Why it happens:**
Using `iframe.srcdoc` (or `src="data:..."`) without the `sandbox` attribute — or with `sandbox` set but including `allow-same-origin` alongside `allow-scripts` — does not actually isolate the frame. A `srcdoc` document is same-origin with its parent by default unless explicitly sandboxed without `allow-same-origin`; combining `allow-scripts` with `allow-same-origin` in the sandbox attribute defeats the sandbox's core protection (the classic "sandbox escape" via those two flags together).

**How to avoid:**
Render the live preview in an iframe with `sandbox="allow-scripts"` and explicitly omit `allow-same-origin`. This lets any generated JS (from Fase JS's "Quan X → Y → Fes Z" rule builder) execute for visual feedback, but the frame cannot access the parent document, its cookies, or the Socket.io client instance. If cross-frame communication is ever needed (e.g., posting computed values back for scoring), use `postMessage` with explicit origin validation on both ends — do not read into the iframe's DOM directly from the parent unless the frame is intentionally non-sandboxed and content is fully server-controlled.

**Warning signs:**
- Preview iframe has no `sandbox` attribute at all.
- Sandbox attribute includes both `allow-scripts` and `allow-same-origin` together.
- Auto-scoring logic reaches directly into the iframe's `contentDocument` from the parent without checking whether the frame is same-origin (which only works if sandboxing was already loosened, defeating the point).

**Phase to address:**
Preview rendering / Fase JS implementation phase, and explicitly verified during the auto-scoring phase (since scoring needs to read the iframe's DOM/computed styles, which forces a specific and easy-to-get-wrong sandboxing tradeoff — see Recovery Strategies below).

### Pitfall 8: No teacher-facing recovery mechanism for a stuck game (single point of failure with no "escape hatch")

**What goes wrong:**
Mid-session, something goes wrong — a team's browser crashes and won't reconnect cleanly, a phase transition doesn't propagate to one screen, or the projected admin view freezes — and the teacher has no way to recover except restarting the entire Node process (losing all team progress) or limping through visibly broken for the remaining 10 minutes of a 15-20 minute session with zero time to spare. In a one-shot live classroom demo, "restart everything" is not an acceptable fallback.

**Why it happens:**
Teams building an admin panel for a short live session tend to focus entirely on the "happy path" controls the spec asks for (start phase, +1 minute, pause/resume, finish) and don't budget for the operational reality: something *will* glitch during a live class with 4-6 different Windows machines, and the admin needs low-cost, low-risk recovery actions that don't nuke the whole match. This is the single biggest asymmetry in this domain: a bug in a typical web app is annoying; a bug in a live 15-minute classroom exercise silently already cost irrecoverable class time even if fixed a minute later.

**How to avoid:**
Add explicit, scoped recovery controls beyond the "happy path," decided during admin-panel design, not bolted on after: (a) per-team "resend current state" button that force-re-emits that team's current phase/timer/board state without touching anyone else; (b) "force phase transition" that re-broadcasts the current phase to all teams (covers the "one screen didn't get the memo" case) without restarting the countdown; (c) a clearly visible per-team connection-status indicator on the admin panel (connected/disconnected/reconnecting) so the teacher immediately sees which specific team is having trouble, rather than guessing; (d) ensure the "Pausar" control genuinely freezes everything so the teacher can pause the whole match to walk over and manually help a stuck team's laptop without the clock (and everyone else's expectations) running out from under them.

**Warning signs:**
- Admin panel spec/design has no per-team resync or "resend state" action.
- No visible per-team connection status indicator for the teacher.
- The only way to recover from "team X's screen is stuck" is closing the browser tab and hoping reconnection logic (Pitfall 1/3) is bulletproof — which it should be, but shouldn't be the *only* line of defense.

**Phase to address:**
Admin panel phase — this should be treated as a first-class requirement, not a stretch goal, given the project's explicit "no margin to repeat" constraint (15-20 min, robustesa crítica).

### Pitfall 9: Overengineering for a scale and lifespan this project will never have

**What goes wrong:**
Time gets spent building a database layer, authentication system, horizontal-scaling-ready Socket.io adapter (Redis pub/sub), or a configurable exercise editor — none of which this v1 needs — while the actually load-bearing robustness work (reconnection, timer authority, scoring fairness, teacher recovery controls above) is under-invested. The project's own "Out of Scope" section already explicitly rules out DB persistence, auth, and a configurable exercise editor, which is the right call — the risk is scope creep re-introducing this complexity mid-build "just in case," or defensive engineering to survive scenarios (server restarts mid-game, multi-instance scaling, thousands of concurrent teams) that will never occur in a single classroom of 4-6 teams for 15-20 minutes.

**Why it happens:**
Developers coming from "production web app" backgrounds default to patterns (persistent DB, session middleware, horizontally-scalable session-sticky Socket.io setups) that are genuinely correct for a multi-tenant, long-lived SaaS product, but are pure waste here. The instinct to "build it right" collides with a spec that is, correctly, optimizing for a single 20-minute live event with one Node process.

**How to avoid:**
Actively hold the line at what's specified: in-memory server state (no DB), no user accounts (admin-assigned team names only), single Node process (no Redis adapter, no multi-instance session affinity beyond the basic Nginx WebSocket-upgrade config from Pitfall 4), one hardcoded exercise (no exercise editor/config UI). Every time a "what if we need to scale this" or "what if the server restarts" thought comes up, explicitly check it against the project's stated scale (4-6 teams + 1 admin, one classroom, one 15-20 min session) before building for it. The correct amount of resilience here is: survive network blips and refreshes (Pitfalls 1-3), not survive process crashes or horizontal scaling.

**Warning signs:**
- Any dependency added for a database (Postgres, Mongo, Redis) "just in case."
- Any authentication/authorization library beyond a simple admin-assigned team name.
- A generic "exercise config" data model when only one exercise (the Plana Model) will ever exist in v1.
- Time spent on a Socket.io Redis adapter or multi-instance deployment topology.

**Phase to address:**
This is a cross-cutting concern to flag at roadmap/phase-planning time itself, not a single implementation phase — each phase's plan should explicitly state "single in-memory Node process, no persistence" as a constraint to prevent scope creep during execution.

## Technical Debt Patterns

| Shortcut | Immediate Benefit | Long-term Cost | When Acceptable |
|----------|-------------------|-----------------|------------------|
| Keying team state by `socket.id` instead of a session token | Faster to write initially | Breaks on every refresh — unacceptable given the "no margin to repeat" constraint | Never for this project |
| Client-side `setInterval` countdown without server sync check | Simple to code first | Visible timer drift across team screens during a projected, competitive game | Never for this project |
| String-comparing `getComputedStyle` output for scoring | Fast to write | Silently marks correct answers wrong (fairness bug, hard to detect without dedicated tests) | Never — but acceptable to start with a documented TODO if a temporary tolerance/format list is used in an early prototype, as long as it's fixed before any real classroom run |
| Skipping the `sandbox` attribute on the preview iframe during early prototyping | Faster to wire up initial preview | Real security gap once JS-rule-builder-generated code executes in preview | Acceptable only in a local-only prototype never exposed to students; must be fixed before any multi-team or deployed test |
| Hardcoding the single Plana Model exercise directly in code (no config layer) | Saves real engineering time | None — this is explicitly the right call per the project's Out of Scope decision, not real debt | Always acceptable for v1 |

## Integration Gotchas

| Integration | Common Mistake | Correct Approach |
|-------------|----------------|-------------------|
| Nginx reverse proxy | Missing `Upgrade`/`Connection` headers or short `proxy_read_timeout`, causing silent long-polling fallback or periodic forced reconnects | Set `proxy_http_version 1.1`, `proxy_set_header Upgrade $http_upgrade`, `proxy_set_header Connection "upgrade"`, and timeouts > Socket.io pingInterval+pingTimeout; verify `transport=websocket` in production, not just localhost |
| Socket.io client reconnection | Assuming reconnection alone restores application state | Pair Socket.io's transport-level reconnect with an explicit application-level resync message sent by the server on every (re)join, keyed by team session token |
| Browser `iframe` for preview | Treating `srcdoc` as inherently safe because "it's just an iframe" | Explicit `sandbox="allow-scripts"` without `allow-same-origin`; use `postMessage` with origin checks for any needed cross-frame data |

## Performance Traps

Given the explicit scale (4-6 teams, one classroom, 15-20 minutes), classic performance traps (DB query N+1, unindexed tables, CDN caching) are not relevant here — there is no database and no meaningful concurrent load. The one real "performance-shaped" trap in this domain:

| Trap | Symptoms | Prevention | When It Breaks |
|------|----------|------------|-----------------|
| Broadcasting full game state to all teams on every tick/update instead of targeted per-team room emits | Unnecessary network chatter, and teams potentially receiving (and needing to filter out) other teams' data | Use Socket.io rooms per team for team-specific state; broadcast only shared/global state (phase, global timer) to all | Irrelevant at 4-6 teams for correctness, but sloppy room usage risks leaking one team's answers to another's preview — a fairness/integrity issue, not a scale issue |

## Security Mistakes

| Mistake | Risk | Prevention |
|---------|------|------------|
| Preview iframe sandboxed with both `allow-scripts` and `allow-same-origin` | Generated content preview can reach into parent page DOM, cookies, or the live Socket.io connection | Use `sandbox="allow-scripts"` only; never combine with `allow-same-origin` for untrusted/generated content |
| No validation of `postMessage` origin if parent/iframe communicate | A malicious or malformed message could be accepted from an unexpected source | Always check `event.origin` and message shape before acting on postMessage payloads |
| Admin-only actions (start phase, force-transition, finalize) not restricted to the admin's socket/connection | A team's browser could, in theory, emit an admin-only event if the server doesn't distinguish roles | Server must validate the emitting socket has the "admin" role (established at connection, e.g., via a distinct admin URL/token) before acting on any admin-scoped event, even though there's no formal auth system |

## UX Pitfalls

| Pitfall | User Impact | Better Approach |
|---------|-------------|-------------------|
| Drag & drop without strong visual snap feedback | Students (with zero technical background) get frustrated dragging a block that "almost" fits but doesn't lock in, wasting precious minutes | Strong snap-to-nearest-valid-slot behavior with clear visual affordance (highlight target before drop, animate snap on drop) — this is explicitly called out in the project's own constraints, treat it as non-negotiable |
| Silent reconnection with no on-screen feedback | A team sees a frozen/blank screen during a brief network blip and doesn't know if it's their fault, panics, or asks the teacher for help mid-game, burning time | Show a lightweight "reconnecting..." indicator during transport reconnects so students understand it's transient |
| Admin panel that doesn't show per-team live status | Teacher can't tell which team is struggling/disconnected without walking around the room, wasting the tight time budget | Per-team status row (name, phase, progress, connection state) visible at a glance, as already scoped in "Vista d'estat/progrés general" |

## "Looks Done But Isn't" Checklist

- [ ] **Reconnection handling:** Often missing a real resync path — verify by registering a team, hard-refreshing mid-phase, and confirming the team lands back exactly where it was (phase, timer, current block/CSS/JS state), not just "socket reconnects."
- [ ] **Timer sync:** Often missing server authority — verify by opening two team browser tabs side by side and confirming their displayed countdown never diverges by more than ~1 second, including across a Pause/Resume/+1 minute admin action.
- [ ] **Auto-scoring fairness:** Often missing normalization — verify with a specific test case where a team produces the visually-correct color/value via a different declared format than the Plana Model reference, and confirm it still scores as correct.
- [ ] **WebSocket transport in production:** Often silently degraded to polling — verify via browser dev tools Network tab against the actual deployed Nginx-fronted domain, not just localhost.
- [ ] **Preview sandboxing:** Often "working" but insecure — verify the `sandbox` attribute is present and does not combine `allow-scripts` with `allow-same-origin`.
- [ ] **Admin recovery controls:** Often only covers the happy path — verify a teacher can resync a single stuck team or force-rebroadcast the current phase without restarting the whole game/server.

## Recovery Strategies

| Pitfall | Recovery Cost | Recovery Steps |
|---------|----------------|------------------|
| Team lost identity/state after refresh (Pitfall 1) | LOW if caught before the live class | Add session-token-based rejoin logic; requires touching the connection handler and client bootstrap, but no data model changes since state is in-memory already |
| Timer drift across clients (Pitfall 2) | LOW-MEDIUM | Replace decrementing counter with absolute `phaseEndsAt`; mostly a client-side rendering change plus one server broadcast field |
| Duplicate sockets per team (Pitfall 3) | LOW | Add an explicit "disconnect prior socket for this token" step in the join handler |
| WebSocket falling back to polling in production (Pitfall 4) | LOW (config-only) but HIGH if discovered live | Fix is a few lines of Nginx config; the real cost is if it's discovered *during* the live classroom session with no time to redeploy — must be verified in a staging/rehearsal run beforehand |
| Scoring fairness bug (Pitfall 5) | MEDIUM | Requires auditing every comparator function and adding normalization + unit tests; higher cost if scoring logic is scattered rather than centralized in one comparator module |
| Unsandboxed preview iframe (Pitfall 7) | LOW | Attribute-only fix, but any parent-code that assumed same-origin access (e.g., reading `contentDocument` directly) needs to switch to postMessage, which is a more involved refactor if it happened |
| No teacher recovery controls (Pitfall 8) | MEDIUM-HIGH if discovered late | Retrofitting admin recovery actions after the core admin/client protocol is built means adding new event types across both ends; cheaper to design in from the start |
